**本周熟悉了 `@Autowired` 依赖注入用法**：在 Service 中通过 `@Autowired` 注入 Mapper 对象，省去了手动获取 Bean 的步骤。

- 掌握了 `@Autowired` 的三种注入方式：字段注入、构造器注入、Setter 注入。
- 实际编码中发现，字段注入虽然写起来最省事，但会导致单元测试必须依赖 Spring 容器启动（慢且笨重）；改用构造器注入后，测试时可以手动 `new` 对象并传入 Mock，单测独立且快很多。
- 顺带区分了 `@Resource`（按名称）和 `@Autowired`（按类型）的差异，明确了 Spring 项目中的使用偏好。

### 1. 先理解：Spring 是如何完成注入的？

不管哪种注入方式，Spring 的注入过程都发生在**容器启动阶段**。容器启动时，会扫描所有 Bean，根据注解（`@Autowired`）去查找匹配的依赖，然后通过反射或构造方法把依赖塞进去。

- **字段注入**：Spring 通过**反射**向 `private` 字段赋值。
- **构造器注入**：Spring 通过**调用构造方法**传入依赖。

在**项目实际运行**时，这两种方式都需要 Spring 容器参与，没什么区别。

------

### 2. 区别发生在“测试”的时候

写单元测试时，目标是**只测当前类的业务逻辑，不测 Spring 容器能否正常启动**。

**① 字段注入的测试（必须启动容器）**

java

```
@Service
public class UserService {
    @Autowired
    private UserMapper userMapper;   // private 字段
}
```



如果你要测这个类，没法直接 `new UserService()`，因为 `userMapper` 是 `private` 的，在外部无法给它赋值（除非用反射）。
所以你只能这么写测试：

java

```
@SpringBootTest   // 启动整个 Spring 容器
class UserServiceTest {
    @Autowired
    private UserService userService; // 从容器里拿已经注入好的对象
}
```



这就是你“必须启动整个项目”的原因——**你不启动容器，`userMapper` 就没有被注入的机会，你拿不到完整的 `UserService` 对象。**

------

**② 构造器注入的测试（无需启动容器）**

java

```
@Service
public class UserService {
    private final UserMapper userMapper;

    @Autowired
    public UserService(UserMapper userMapper) {   // 依赖通过构造器传入
        this.userMapper = userMapper;
    }
}
```



测试时，可以**手动传一个 Mock 对象进去**，完全不需要 Spring：

java

```
class UserServiceTest {
    @Test
    void test() {
        UserMapper mockMapper = Mockito.mock(UserMapper.class);  // 模拟依赖
        UserService service = new UserService(mockMapper);        // 直接 new
        // 然后测你的业务逻辑
    }
}
```



这里 `UserService` 的构造方法是公开的（或至少是可访问的），能在测试代码里直接调用它，把依赖“塞”进去。**整个过程没有 Spring 容器参与**，所以启动非常快。
