## 二、核心学习内容

### 2.1 PageHelper 分页插件原理

PageHelper 是基于 MyBatis 拦截器机制实现的分页插件，通过拦截 Executor 的 query 方法，在执行 SQL 前自动拼接分页语句，实现了对业务代码的零侵入。

**核心特性**：
- **零侵入**：无需修改已有 SQL，只需在查询前调用 `PageHelper.startPage()` 即可
- **线程安全**：分页参数通过 `ThreadLocal` 传递，自动清理
- **多数据库支持**：自动识别数据库方言，生成对应的分页 SQL

### 2.2 分页查询完整实现

#### 2.2.1 Service 层

Service 层负责接收分页参数，调用 PageHelper 启动分页，并返回分页结果对象。

**Service 层**

```java
@Service
public class UserService {
    @Autowired
    private UserMapper userMapper;

    public PageInfo<User> listUsers(Integer pageNum, Integer pageSize, String keyword) {
        PageHelper.startPage(pageNum, pageSize);
        List<User> list = userMapper.selectByKeyword(keyword);
        return new PageInfo<>(list);
    }
}
```

**关键点说明**：
- `PageHelper.startPage(pageNum, pageSize)` 必须在 Mapper 查询之前调用，且仅对紧跟其后的一条查询生效
- `PageInfo<T>` 封装了分页结果，包含 `total`（总记录数）、`pages`（总页数）、`list`（当前页数据）等字段

#### 2.2.2 Mapper 接口

**Mapper 接口**

```java
@Mapper
public interface UserMapper {
    List<User> selectByKeyword(String keyword);
}
```

#### 2.2.3 Mapper XML

Mapper XML 中实现带条件判断的动态 SQL，配合 PageHelper 自动完成分页。

**Mapper XML**

```xml
<select id="selectByKeyword" resultType="User">
    SELECT * FROM user
    <where>
        <if test="keyword != null and keyword != ''">
            username LIKE CONCAT('%', #{keyword}, '%')
        </if>
    </where>
</select>
```

**技术要点**：
- `<where>` 标签自动处理开头的 `AND`/`OR`，且在没有条件时不生成 `WHERE` 子句
- `CONCAT` 函数拼接模糊查询的通配符，实现关键字搜索

#### 2.2.4 Controller 层

**Controller 层**

```java
@RestController
@RequestMapping("/users")
public class UserController {
    @Autowired
    private UserService userService;

    @GetMapping
    public Result listUsers(
            @RequestParam(defaultValue = "1") Integer pageNum,
            @RequestParam(defaultValue = "10") Integer pageSize,
            @RequestParam(required = false) String keyword) {
        PageInfo<User> pageInfo = userService.listUsers(pageNum, pageSize, keyword);
        return Result.success(pageInfo);
    }
}
```

**设计要点**：
- `defaultValue` 设置分页默认值：第一页、每页 10 条
- `required = false` 使关键字参数可选，支持不带搜索条件查询全量数据
- 统一返回封装类 `Result` 保证接口响应格式一致

### 2.3 进阶配置

#### 2.3.1 带排序的分页查询

```java
PageHelper.startPage(pageNum, pageSize, "id DESC");
// 或者
PageHelper.startPage(pageNum, pageSize);
PageHelper.orderBy("id DESC");
```

`orderBy` 参数直接写入 SQL 的 `ORDER BY` 子句，避免前端拼接排序字段时的 SQL 注入风险。

#### 2.3.2 不查总记录数（提升性能）

```java
PageHelper.startPage(pageNum, pageSize, false);
```

当数据量较大或仅需展示数据而无需总页数时，可跳过 `COUNT` 查询，显著提升查询性能。

#### 2.3.3 前端对接的 JSON 格式

**前端拿到的 JSON**

```json
{
  "code": 200,
  "data": {
    "pageNum": 1,
    "pageSize": 10,
    "total": 100,
    "pages": 10,
    "list": [
      { "id": 1, "username": "zhangsan" },
      { "id": 2, "username": "lisi" }
    ]
  }
}
```

**PageInfo 核心字段说明**：

| 字段     | 说明              |
| -------- | ----------------- |
| pageNum  | 当前页码          |
| pageSize | 每页显示条数      |
| total    | 总记录数          |
| pages    | 总页数            |
| list     | 当前页数据集合    |

### 2.4 @Transactional 事务管理

#### 2.4.1 声明式事务核心原理

Spring 的声明式事务基于 **AOP 代理机制** 实现，通过 `@Transactional` 注解在方法执行前后自动开启和提交/回滚事务，将事务管理与业务逻辑解耦。

**核心特性**：
- **声明式配置**：注解驱动，无需编程式事务管理，降低代码侵入性
- **AOP 代理拦截**：Spring 为标注了 `@Transactional` 的 Bean 生成代理对象，在方法调用前后织入事务逻辑
- **默认回滚策略**：仅对 `RuntimeException` 和 `Error` 自动回滚，受检异常（Checked Exception）默认不回滚

**基础使用示例**：

```java
@Service
public class UserService {
    @Autowired
    private UserMapper userMapper;
    @Autowired
    private OrderMapper orderMapper;

    @Transactional
    public void createUserWithOrder(User user, Order order) {
        userMapper.insert(user);          // 插入用户
        orderMapper.insert(order);        // 插入订单
        // 任一操作失败，整个方法回滚
    }
}
```

**关键注意点**：
- `@Transactional` 加在 **public 方法** 上才生效，private 方法上的注解会被忽略
- **同类自调用失效**：同一个类中方法 A（无事务）直接调用方法 B（有 `@Transactional`），事务不生效，因为绕过了代理对象
- 可通过 `@Autowired` 注入自身或使用 `AopContext.currentProxy()` 解决自调用问题

#### 2.4.2 事务传播行为（Propagation）

事务传播行为定义了 **一个事务方法被另一个事务方法调用时，事务如何传播**。Spring 共定义了七种传播行为：

| 传播行为                   | 说明                                                         | 典型应用场景                       |
| -------------------------- | ------------------------------------------------------------ | ---------------------------------- |
| **REQUIRED**（默认）       | 如果当前存在事务则加入，否则新建事务                         | 最常用，适用于绝大多数增删改场景   |
| **REQUIRES_NEW**           | 始终新建事务，如果当前存在事务则挂起                         | 日志记录、审计，不受外层事务影响   |
| **SUPPORTS**               | 如果当前存在事务则加入，否则以非事务方式运行                 | 查询方法，有事务就参与，没有也无妨 |
| **NOT_SUPPORTED**          | 以非事务方式运行，如果当前存在事务则挂起                     | 不需要事务的耗时操作，避免占用连接 |
| **MANDATORY**              | 必须在已有事务中运行，否则抛出异常                           | 强制调用方必须开启事务             |
| **NEVER**                  | 必须以非事务方式运行，如果存在事务则抛出异常                 | 禁止在事务中调用的方法             |
| **NESTED**                 | 如果当前存在事务，则创建嵌套事务（Savepoint 机制），外层回滚会影响内层 | 批量处理中实现部分失败部分提交     |

**代码示例**：

```java
@Service
public class OrderService {
    @Autowired
    private OrderMapper orderMapper;
    @Autowired
    private LogService logService;

    // 默认 REQUIRED：下单与扣库存在同一事务中
    @Transactional
    public void placeOrder(Order order) {
        orderMapper.insert(order);
        inventoryService.deductStock(order);  // 扣库存失败则下单也回滚
    }
}

@Service
public class LogService {

    // REQUIRES_NEW：日志独立事务，不受外层回滚影响
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void saveErrorLog(String message) {
        logMapper.insert(new ErrorLog(message));
        // 即使外层下单事务回滚，日志也会被保留
    }
}
```

#### 2.4.3 事务的四大核心属性

`@Transactional` 除了 `propagation` 还支持以下核心配置：

```java
@Transactional(
    propagation = Propagation.REQUIRED,      // 传播行为
    isolation = Isolation.READ_COMMITTED,    // 隔离级别
    timeout = 30,                            // 超时时间（秒），默认 -1 不超时
    readOnly = false,                        // 只读事务（true 时优化查询性能）
    rollbackFor = Exception.class,           // 指定触发回滚的异常类型
    noRollbackFor = IllegalArgumentException.class  // 指定不触发回滚的异常
)
public void businessMethod() { ... }
```

**常用属性说明**：

| 属性          | 说明                                                | 建议                                                       |
| ------------- | --------------------------------------------------- | ---------------------------------------------------------- |
| `propagation` | 事务传播行为                                        | 写操作用 REQUIRED，独立日志用 REQUIRES_NEW                  |
| `isolation`   | 事务隔离级别（DEFAULT / READ_UNCOMMITTED / READ_COMMITTED / REPEATABLE_READ / SERIALIZABLE） | 一般用默认（跟随数据库），高一致性场景用 READ_COMMITTED     |
| `timeout`     | 事务超时秒数，超时后自动回滚                        | 长事务设置合理超时防止锁表                                  |
| `readOnly`    | 标记为只读，底层可做查询优化                        | 纯查询方法设置为 `true`，提升性能                           |
| `rollbackFor` | 指定哪些异常触发回滚                                | 若需受检异常也回滚，设置 `rollbackFor = Exception.class`    |

#### 2.4.4 常见事务失效场景

**场景一：同类自调用**

```java
@Service
public class UserService {
    // ❌ 错误：直接调用同类中带 @Transactional 的方法，事务不生效
    public void batchCreate(List<User> users) {
        for (User u : users) {
            this.createUser(u);  // this 引用的是原始对象，不是代理
        }
    }

    @Transactional
    public void createUser(User user) {
        userMapper.insert(user);
    }

    // ✅ 正确：通过注入代理对象调用
    @Autowired
    private UserService self;

    public void batchCreate(List<User> users) {
        for (User u : users) {
            self.createUser(u);  // 通过代理调用，事务生效
        }
    }
}
```

**场景二：方法非 public**

```java
// ❌ 错误：private 方法上的 @Transactional 被忽略
@Transactional
private void doInsert(User user) {
    userMapper.insert(user);
}
```

**场景三：异常被捕获吞掉**

```java
// ❌ 错误：异常被 try-catch 吞掉，事务不回滚
@Transactional
public void createUser(User user) {
    try {
        userMapper.insert(user);
        int i = 1 / 0;  // 抛出 ArithmeticException
    } catch (Exception e) {
        log.error("error", e);
        // 没有重新抛出，Spring 感知不到异常，事务提交
    }
}

// ✅ 正确：捕获后重新抛出 RuntimeException
@Transactional
public void createUser(User user) {
    try {
        userMapper.insert(user);
    } catch (Exception e) {
        log.error("error", e);
        throw new RuntimeException("操作失败", e);  // 触发回滚
    }
}
```

**场景四：@Transactional 加在 Controller 层**

```java
// ❌ 不推荐：事务应放在 Service 层，Controller 职责是参数校验和视图转发
@RestController
public class UserController {
    @Transactional  // 事务边界过宽，应下沉到 Service
    @PostMapping("/users")
    public Result createUser(@RequestBody User user) { ... }
}
```

## 四、问题与解决方案

### 4.1 PageHelper 分页不生效

**问题**：在 Mapper 查询后执行了其他查询操作，导致 PageHelper 分页参数被后续查询消费，目标查询未分页。

**解决**：确保 `PageHelper.startPage()` 与目标 Mapper 查询方法之间无其他数据库查询操作；如需多次分页查询，每次查询前重新调用 `startPage()`。

### 4.2 分页总数查询性能瓶颈

**问题**：大表查询时 COUNT 操作耗时过长，影响接口响应速度。

**解决**：对于不需要展示总页数的场景（如滚动加载），使用 `PageHelper.startPage(pageNum, pageSize, false)` 跳过 COUNT 查询，直接返回指定页的数据。

### 4.3 同类方法自调用导致事务失效

**问题**：在 Service 类中，方法 A（无 `@Transactional`）直接通过 `this.方法B()` 调用方法 B（有 `@Transactional`），事务未生效。

**原因**：Spring 事务基于 AOP 动态代理实现，`this` 引用的是原始对象而非代理对象，调用绕过了代理拦截。

**解决**：通过 `@Autowired` 注入自身 Bean，使用代理对象调用事务方法；或将事务方法抽取到独立的 Service 类中，确保调用经过代理。

### 4.4 异常被 try-catch 吞掉导致数据不一致

**问题**：事务方法中捕获了异常但未重新抛出，Spring 感知不到异常发生，事务正常提交，导致部分成功部分失败的数据不一致问题。

**解决**：在 catch 块中重新抛出 `RuntimeException`；或调用 `TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()` 手动标记回滚。
